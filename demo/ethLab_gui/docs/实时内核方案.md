# EtherCAT 主站实时性方案选型

当前系统:
- 发行版: Ubuntu 22.04
- 内核: `6.8.0-107-generic` (`PREEMPT_DYNAMIC`, HWE 版)
- 硬件: Lenovo ThinkBook 16p Gen4
- 目标: IgH EtherCAT 主站,周期 1000 µs,CSP/PP 多模式控制

---

## 方案对比一览

| 方案 | 改动量 | 预期抖动(1ms 周期) | 需要重启 | 稳定性 | 推荐度 |
|---|---|---|---|---|---|
| A. 仅 SCHED_FIFO + mlockall (已做) | 代码 | 50~300 µs | 否 | 高 | ★★★★ |
| B. A + `preempt=full` 动态抢占 | grub 参数 | 30~150 µs | 是 | 高 | ★★★★★ |
| C. B + `sched_rt_runtime_us=-1` + 绑核 | sysctl/启动 | 20~80 µs | 否 | 高 | ★★★★★ |
| D. C + `isolcpus` CPU 隔离 | grub 参数 | 15~50 µs | 是 | 高 | ★★★★ |
| E. PREEMPT_RT 内核 | 换内核 | 5~20 µs | 是 | 视补丁成熟度 | ★★★ |
| F. Xenomai / EtherLab userspace 调度器 | 大改 | < 10 µs | 是 | 中 | ★★ |

判定原则:
- 1ms 周期、单轴/少轴:**方案 B 足够**。
- 多轴高精度(插补/联动):**方案 C 或 D**。
- 微秒级刚性需求(< 50 µs 抖动):**方案 E**。

## 方案 A:仅软件侧(已经生效)

`EcWorker::run()` 开头已加:

```cpp
pthread_setschedparam(pthread_self(), SCHED_FIFO, &sp /*prio=80*/);
mlockall(MCL_CURRENT | MCL_FUTURE);
```

### 前置条件

运行时 **必须 `sudo`**,或给可执行文件 capability:

```bash
sudo setcap cap_sys_nice,cap_ipc_lock+ep ./ethLab_gui
```

且当前用户的 `ulimit -r` ≥ 80:

```bash
ulimit -r            # 查看
# 若不足,编辑 /etc/security/limits.conf 加:
# ruio  -  rtprio  99
# ruio  -  memlock unlimited
```

### 验证

日志里应出现 `RT 线程已设置 SCHED_FIFO priority=80`,每秒 `周期诊断 ... overrun=0/1000` 常态。

---

## 方案 B:启用 `preempt=full`

`PREEMPT_DYNAMIC` 允许运行时切换抢占强度,默认是 `voluntary`,对 RT 不够激进。

### B1. 临时切换(重启失效)

```bash
# 查看可选/当前值(括号里是当前)
cat /sys/kernel/debug/sched/preempt
# 输出例: none voluntary (full)

sudo sh -c 'echo full > /sys/kernel/debug/sched/preempt'
```

### B2. 永久生效

```bash
sudo nano /etc/default/grub
```

把这行改成(加 `preempt=full`):

```
GRUB_CMDLINE_LINUX_DEFAULT="quiet splash preempt=full"
```

然后:

```bash
sudo update-grub
sudo reboot
```

验证:

```bash
cat /sys/kernel/debug/sched/preempt        # (full)
cat /sys/kernel/debug/sched/preempt_mode   # 若存在
```

## 方案 C:放行 RT 运行时间 + 绑核

### C1. 放行 RT 时间片

Linux 默认 `/proc/sys/kernel/sched_rt_runtime_us = 950000` —— 意思是每 1 秒 RT 任务最多占 950ms,剩下 50ms 会被强制让给普通进程,偶尔引发几 ms 的周期抖动。

```bash
# 临时
sudo sysctl -w kernel.sched_rt_runtime_us=-1

# 永久
echo 'kernel.sched_rt_runtime_us=-1' | sudo tee /etc/sysctl.d/99-rt.conf
sudo sysctl --system
```

> 风险: RT 线程若死循环会把一个 CPU 吃满,系统没有保护。本程序有 `clock_nanosleep` 每周期让出,安全。

### C2. 绑核运行

查看 CPU 数量:

```bash
nproc            # 例如 16
lscpu | grep -i 'NUMA\|core\|socket'
```

选一个 **最远离 0 号核** 的(0 号常被系统中断占用),例如 3:

```bash
sudo taskset -c 3 ./ethLab_gui
```

也可以在程序内部绑定(更稳):

```cpp
cpu_set_t set; CPU_ZERO(&set); CPU_SET(3, &set);
pthread_setaffinity_np(pthread_self(), sizeof(set), &set);
```

## 方案 D:CPU 隔离(独占核)

让某个核完全不被普通进程/定时器/RCU 打扰,RT 线程独占。

编辑 `/etc/default/grub`:

```
GRUB_CMDLINE_LINUX_DEFAULT="quiet splash preempt=full isolcpus=3 nohz_full=3 rcu_nocbs=3"
```

然后:

```bash
sudo update-grub && sudo reboot
```

效果:CPU 3 从此不会被调度普通进程,必须用 `taskset -c 3` 或 `sched_setaffinity` 主动绑定才用。

验证:

```bash
cat /sys/devices/system/cpu/isolated        # 3
cat /sys/devices/system/cpu/nohz_full       # 3
```

另外把网卡中断也移开 CPU 3:

```bash
# 假设 ethercat 绑定的网卡是 eno1
grep eno1 /proc/interrupts
# 看到中断号 N 后:
echo 7 | sudo tee /proc/irq/<N>/smp_affinity   # 只用 CPU0~2
```

## 方案 E:安装 PREEMPT_RT 内核

Ubuntu 22.04 默认仓库没有 RT 内核,有三条路径,按推荐度排序。

---

### E1. 路线一:Ubuntu Pro Real-time Kernel(最省心,个人免费)

Canonical 官方 RT 内核。注册 Ubuntu Pro 后个人免费可开 5 台机器。

#### 步骤

```bash
# 1. 浏览器注册 https://ubuntu.com/pro/dashboard,复制 token

# 2. 绑定本机
sudo pro attach <你的token>

# 3. 启用 realtime 内核(自动装 linux-realtime 包)
sudo pro enable realtime-kernel

# 4. 重启选 realtime 内核
sudo reboot
```

#### 验证

```bash
uname -a
# 期望: Linux xxx ... -realtime ... PREEMPT_RT ...
cat /sys/kernel/realtime        # 输出 1 即真正 RT
```

#### 回退

```bash
sudo pro disable realtime-kernel
# 重启选旧内核
```

#### 注意

- 版本锁在 22.04 对应的 RT 分支(当前 6.8-realtime),跟随 LTS 生命周期。
- 与 `linux-lowlatency` 互斥,若已装低延迟内核先 `sudo apt remove linux-lowlatency*`。
- **IgH `ec_master` / `ec_generic` 模块必须针对新内核重编**(见下方 E4)。

---

### E2. 路线二:第三方 PPA

非官方维护,稳定性不可控,不建议生产。略。

---

### E3. 路线三:自编译 PREEMPT_RT 内核(无需订阅,完全掌控)

#### E3.1 选版本

补丁列表: https://mirrors.edge.kernel.org/pub/linux/kernel/projects/rt/

推荐选 LTS,如 `6.6.y-rt`。

> 自 **Linux 6.12** 起 `PREEMPT_RT` 已合并主线,不需补丁,直接 `CONFIG_PREEMPT_RT=y`。但 6.12 还没进 22.04 默认仓库,需自己编或上 mainline PPA。

#### E3.2 安装依赖

```bash
sudo apt update
sudo apt install -y build-essential libncurses-dev bison flex libssl-dev \
                    libelf-dev bc dwarves fakeroot rsync ccache
```

#### E3.3 下载源码 + 补丁

以 `6.6.55-rt45` 为例:

```bash
mkdir -p ~/kernel-rt && cd ~/kernel-rt

wget https://cdn.kernel.org/pub/linux/kernel/v6.x/linux-6.6.55.tar.xz
wget https://mirrors.edge.kernel.org/pub/linux/kernel/projects/rt/6.6/older/patch-6.6.55-rt45.patch.xz

tar xf linux-6.6.55.tar.xz
cd linux-6.6.55
xzcat ../patch-6.6.55-rt45.patch.xz | patch -p1
```

用 6.12+ 主线时跳过 patch 步。

#### E3.4 配置

```bash
# 以当前内核配置为起点
cp /boot/config-$(uname -r) .config
make olddefconfig

# 启用 RT
./scripts/config --enable PREEMPT_RT
./scripts/config --disable PREEMPT_DYNAMIC
./scripts/config --disable PREEMPT_VOLUNTARY
./scripts/config --enable HIGH_RES_TIMERS

# 关掉 Ubuntu 签名密钥(否则报 canonical-certs.pem 缺失)
./scripts/config --disable SYSTEM_TRUSTED_KEYS
./scripts/config --disable SYSTEM_REVOCATION_KEYS
./scripts/config --set-str SYSTEM_TRUSTED_KEYS ""
./scripts/config --set-str SYSTEM_REVOCATION_KEYS ""

# 关 debug 减少体积
./scripts/config --disable DEBUG_INFO
./scripts/config --enable  DEBUG_INFO_NONE

make olddefconfig
```

可选用图形菜单再核对:

```bash
make menuconfig
# General setup → Preemption Model → (*) Fully Preemptible Kernel (Real-Time)
# Timers subsystem → High Resolution Timer Support → [*]
```

#### E3.5 编译打包

```bash
make -j$(nproc) bindeb-pkg LOCALVERSION=-rt
```

耗时约 30~90 分钟。完成后 `~/kernel-rt/` 下出现:

```
linux-image-6.6.55-rt+_*.deb
linux-headers-6.6.55-rt+_*.deb
linux-libc-dev_*.deb
```

#### E3.6 安装

```bash
cd ~/kernel-rt
sudo dpkg -i linux-image-6.6.55-rt*.deb linux-headers-6.6.55-rt*.deb
sudo update-grub
sudo reboot        # 开机 GRUB → Advanced options → 选 6.6.55-rt
```

#### E3.7 验证

```bash
uname -a                                       # PREEMPT_RT 字样
cat /sys/kernel/realtime                       # 1
grep PREEMPT /boot/config-$(uname -r)          # CONFIG_PREEMPT_RT=y
```

### E4. 装完 RT 内核必须重装 IgH 模块

任何 E1/E3 路线换完内核后都要做:

```bash
# 停服务
sudo systemctl stop ethercat

# 进入之前下载的 ethercat 源码
cd /path/to/ethercat-source    # 例如 /usr/src/ethercat-xxx

# 卸载旧版(如果之前 make install 过)
sudo make uninstall

# 针对新内核重配
./configure --enable-generic \
            --with-linux-dir=/usr/src/linux-headers-$(uname -r)
#  native 网卡驱动慎用,RT 下推荐 --enable-generic

make -j$(nproc)
sudo make modules_install install
sudo depmod -a

# 启动
sudo systemctl start ethercat
sudo systemctl status ethercat

# 用户侧库如果用 /opt/etherlab 或 /usr/local/etherlab,
# 本程序 CMakeLists 指向了 /usr/local/etherlab/{include,lib},
# 换内核不影响用户库,一般无需重编本程序。
```

> RT 下推荐 `--enable-generic`:native 专用驱动(e1000e-native 等)与 RT 补丁偶有不兼容。

---

### E5. 程序侧需要改吗?

**不用改**。`EcWorker::run()` 已包含:
- `pthread_setschedparam(SCHED_FIFO, 80)`
- `mlockall(MCL_CURRENT | MCL_FUTURE)`

RT 内核下这些调用会自动发挥更强效果。建议附加:
- 绑核: `sudo taskset -c 3 ./ethLab_gui`
- `sysctl kernel.sched_rt_runtime_us=-1`
- grub: `isolcpus=3 nohz_full=3 rcu_nocbs=3`

### E6. 预期收益

| 指标 | PREEMPT_DYNAMIC | PREEMPT_RT |
|---|---|---|
| 1ms 周期 max | 100~500 µs | 10~30 µs |
| 1ms 周期 jitter | 几十 µs | 个位 µs |
| 典型 overrun | 偶发 | ≈ 0 |
| 多轴插补能力 | 有限 | 优秀 |
| 最小可用周期 | ~500 µs | ~125 µs |

### E7. 常见坑

1. **编译报 `No rule to make target 'debian/canonical-certs.pem'`**
   → E3.4 的 `SYSTEM_TRUSTED_KEYS` 没清空,执行 `./scripts/config --set-str SYSTEM_TRUSTED_KEYS ""` 后 `make olddefconfig` 再编。

2. **GRUB 进不去新内核 / 黑屏**
   → 开机按 Shift/ESC 进 GRUB 选旧内核。检查 `/boot/initrd.img-*-rt` 是否生成;缺失就 `sudo update-initramfs -c -k <版本>`。

3. **IgH 模块 `insmod` 报 "Unknown symbol" 或 "Invalid module format"**
   → 没重新 `make modules_install`,或 native 驱动与 RT 冲突,换 `--enable-generic`。

4. **RT 内核下桌面更卡**
   → 显卡/蓝牙等中断亲和度太广。找到对应 IRQ:`grep -E 'i915|nvidia|xhci' /proc/interrupts`,然后 `echo <mask> | sudo tee /proc/irq/<N>/smp_affinity` 把它们从 RT 核上赶走。

5. **Ubuntu Pro 路线 `pro enable realtime-kernel` 失败**
   → `pro status` 看 attach 状态;删除已有 lowlatency 内核再试。

6. **`PREEMPT_RT` 下偶发软锁死警告**
   → 多为网卡中断线程优先级太低。`chrt -f -p 50 $(pidof irq/<N>-eno1)` 暂时提权观察。

## 快速起步(推荐步骤)

按顺序做,边做边看 `周期诊断` 日志里的 `max` 与 `overrun`:

1. **先确认 A 生效**:`sudo ./ethLab_gui`,日志有 `SCHED_FIFO priority=80`。
2. **一键做 B + C1**:

   ```bash
   # B: 永久
   sudo sed -i 's|GRUB_CMDLINE_LINUX_DEFAULT="quiet splash"|GRUB_CMDLINE_LINUX_DEFAULT="quiet splash preempt=full"|' /etc/default/grub
   sudo update-grub

   # C1: 永久
   echo 'kernel.sched_rt_runtime_us=-1' | sudo tee /etc/sysctl.d/99-rt.conf

   sudo reboot
   ```

3. 重启后运行,记录每秒 `周期诊断`。若 `max` < 200µs、`overrun=0`:**达标,不用再折腾**。
4. 若仍有 overrun,再做 **C2 绑核**(`sudo taskset -c 3 ./ethLab_gui`)。
5. 仍不够再上 **D isolcpus**。
6. 以上都不够(极少见)再考虑 E。

## 诊断日志解读速查

```
周期诊断 目标=1000us 实测 avg=1000.02us min=998.76us max=1003.21us 抖动=4.45us | 负载 avg=42.18us max=91.60us | overrun=0/1000
```

- `avg` 偏离目标 > 5µs: 调度被拖,不正常。
- `max` > 2×目标: 丢周期,会引发 EtherCAT 从站掉 OP。
- `负载 avg` > 周期 50%: RT 负载过重,可能算法里有阻塞操作。
- `overrun > 0`: 至少有一个周期超过 1.5×目标,应定位原因(通常是系统中断或日志 I/O)。

## 常见问题

**Q: 加了 `preempt=full` 后桌面/图形会卡吗?**
A: 不会明显感觉。桌面进程仍是 `SCHED_OTHER`,只有你的 RT 线程会抢占它们,占比极低。

**Q: `sched_rt_runtime_us=-1` 是否危险?**
A: 仅在 RT 线程自身没死循环时安全。本项目 `clock_nanosleep` 会主动让出,放心。

**Q: 装 PREEMPT_RT 后 IgH 模块需要改吗?**
A: 需要重新针对新内核编译 `ec_master` / `ec_generic` 模块。命令与你当前装 IgH 时相同。

**Q: `isolcpus` 是否浪费 CPU?**
A: 只要本项目运行就不浪费(RT 线程独占);不运行时该核确实空闲,对桌面不影响性能。
