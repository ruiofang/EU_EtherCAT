# EU EtherCAT 机械臂问题总结

## 1. 问题现象

机械臂程序启动 EtherCAT 主站时，7 个 EU 电机从站不能稳定地全部进入 OP。典型日志如下：

```text
#0 AL=0x08 operational=1
#1 AL=0x08 operational=1
...
#3 AL=0x04 operational=0
```

失败位置不是固定的：一次可能是 3 号从站停在 SAFE-OP，下一次可能变成 5 号从站。这说明问题不是某一台电机固定损坏，也不像是某个固定位置的 PDO 映射错误。

## 2. 已确认的直接原因

IgH 内核日志给出了明确错误：

```text
Failed to set OP state, slave refused state change (SAFEOP + ERROR)
AL status message 0x001B: "Sync manager watchdog"
Datagram ... was SKIPPED
Datagram UNMATCHED
Domain working counter changed from 21/21 to 0/21
```

`0x001B Sync manager watchdog` 表示从站在规定时间内没有持续收到有效的输出过程数据，因此拒绝 SAFE-OP → OP 转换。

程序日志中的：

```text
master响应=7
domainWC=21/2
```

说明：

- 7 个从站都能响应，物理链路没有完全断开；
- PDO 注册数量正确，完整 Working Counter 为 21；
- 但运行过程中 Working Counter 曾突然降为 0，表明周期帧发生了明显中断。

## 3. USB 网卡问题

当前 EtherCAT 使用的网卡信息为：

```text
interface: eth0
driver: r8152
hardware: Realtek RTL8153B USB Ethernet
bus-info: usb-0000:00:14.0-2.1.4.1
speed: 100Mb/s Full Duplex
```

该网卡通过多级 USB 路径连接。USB 网卡的数据传输会受到以下因素影响：

- USB 主控调度和批量传输延迟；
- USB Hub 和同一总线其他设备争用；
- r8152 驱动缓存与中断合并；
- USB 自动节能或挂起；
- Linux 普通网络驱动的不确定调度延迟；
- GRO/GSO/TSO 等网络聚合与卸载功能。

普通 TCP/IP 通信通常能容忍毫秒甚至更长时间的延迟，但 EtherCAT 的 Sync Manager watchdog 要求过程数据持续、确定地到达。USB 网卡偶发的一次长延迟就可能让某台从站进入 SAFE-OP + ERROR。

因此，即使平均帧率正常、链路显示 `UP`、丢包比例看起来很低，也不能说明该网卡适合实时 EtherCAT。

## 4. 周期策略还是网卡设置

这次故障需要区分“程序周期策略”和“网卡/系统实时性”两类问题。

### 4.1 周期策略

程序曾尝试在启动阶段将通信间隔从 1000 μs 调整到 500 μs，并延长等待时间、调用 `ecrt_master_reset()`。

这些措施只能改善正常状态转换的等待过程，不能修复底层周期帧中断：

- 延长 10 秒到 25 秒不会消除 SM watchdog；
- `ecrt_master_reset()`只会重新运行从站配置状态机；
- 500 μs 会提高 USB 网卡的收发压力，未必比 1000 μs稳定；
- 如果 Working Counter 会从 21 突然降到 0，问题已经超出单纯等待策略的范围。

机械臂正常运行周期仍建议使用 1000 μs，但前提是网卡、驱动和实时调度能够稳定支持该周期。

### 4.2 网卡和系统设置

当前日志还显示：

```text
SCHED_FIFO 设置失败(rc=1)
```

这表示 EtherCAT 周期线程运行在普通 `SCHED_OTHER/TS` 调度策略下，而不是设定的 `SCHED_FIFO priority=80`。桌面程序、图形界面、磁盘或其他系统负载可能抢占周期线程。

应优先使用实时权限启动：

```bash
cd /home/ruio/data/123/国家电网/EU_EtherCAT
sudo -E ./EU_robot_arm_gui/build/EU_robot_arm_gui
```

日志中应出现：

```text
RT 线程已设置 SCHED_FIFO priority=80
```

可临时关闭网卡聚合/分段卸载进行对比测试：

```bash
sudo ethtool -K eth0 gro off gso off tso off
```

检查设置：

```bash
ethtool -i eth0
ethtool -k eth0
ethtool eth0
```

这些设置只能用于验证和缓解，不能让 USB 网卡具备 PCIe 实时网卡的确定性。

## 5. 推荐解决方案

### 临时测试方案

1. USB 网卡直接连接电脑，不经过扩展坞或多级 USB Hub。
2. 拔掉同一 USB 控制器上的大流量设备。
3. 使用 `sudo -E` 启动程序，确认 SCHED_FIFO 设置成功。
4. 关闭 GRO/GSO/TSO 后重复启动测试。
5. 观察内核日志是否仍出现 `SKIPPED`、`UNMATCHED` 和 Working Counter 降为 0。

查看相关日志：

```bash
sudo journalctl -k --since "5 minutes ago" --no-pager | grep -E "EtherCAT|watchdog|SKIPPED|UNMATCHED"
```

### 正式运行方案

机械臂正式运行建议更换为：

- Intel I210/I211 等独立 PCIe 千兆网卡；
- 不经过 USB Hub 或 USB 转网口；
- 优先使用 IgH 支持的 native EtherCAT 网卡驱动；
- 配合 PREEMPT_RT 内核、CPU 隔离及 IRQ 亲和性设置。

更换网卡后应进行长时间测试，确认：

```text
overrun 长期为 0
Working Counter 始终完整
无 Datagram SKIPPED
无 Datagram UNMATCHED
无 AL 0x001B Sync manager watchdog
```

## 6. 安全注意事项

不建议通过禁用 Sync Manager watchdog 来绕过启动失败。禁用 watchdog 后，从站可能在主站通信中断时继续保持上一次控制输出。对于机械臂，这可能导致电机持续运动、保持错误力矩或无法及时失能。

当前 USB 网卡已经出现接近一秒的过程数据中断。即使偶尔能够成功进入 OP，也不代表可以安全执行机械臂动作。完成网卡和实时调度整改前，仅应在低速、无负载、急停有效且人员远离运动范围的条件下测试。

## 7. 最终结论

本次启动失败的直接错误是 `AL 0x001B Sync manager watchdog`。根本诱因是 Realtek `r8152` USB 网卡产生不确定的周期帧延迟，同时程序没有获得 SCHED_FIFO 实时调度权限。

PDO 映射和从站识别基本正常；继续延长启动等待或反复重置主站不能从根本上解决。正式解决方向是使用实时权限运行，并将 USB 网卡更换为适合 EtherCAT 的独立 PCIe 网卡。
